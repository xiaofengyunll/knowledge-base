# 数据库知识要点（第一节课）

---

## 一、数据库基础概念

### 1.1 基本认识

#### 数据、数据库、数据库管理系统、数据库系统的区别

| 概念 | 说明 | 类比 |
|------|------|------|
| **数据（Data）** | 描述事物的符号记录，可以是数字、文字、图像、音频等 | 书的内容 |
| **数据库（DB）** | 长期存储在计算机内、有组织、可共享的大量数据的集合 | 图书馆的书架 |
| **数据库管理系统（DBMS）** | 管理数据库的软件，提供数据定义、操作、控制等功能（如 MySQL、Oracle） | 图书馆管理系统 |
| **数据库系统（DBS）** | 由数据库、DBMS、应用程序、用户、数据库管理员（DBA）组成的整体 | 整个图书馆体系（书+管理员+系统+读者） |

四者关系：**数据** 存储在 **数据库** 中，**DBMS** 管理数据库，三者加上用户和应用程序构成 **数据库系统**。

#### 文件系统与数据库系统的区别

| 对比维度 | 文件系统 | 数据库系统 |
|----------|----------|------------|
| 数据结构化 | 无结构或简单结构 | 高度结构化（表、关系） |
| 数据共享性 | 差，文件各自独立 | 好，多用户可同时访问 |
| 数据独立性 | 差，程序与数据紧耦合 | 好，三级模式两级映像 |
| 冗余度 | 高，各文件独立存储 | 低，数据统一管理 |
| 一致性 | 难以保证 | 由 DBMS 通过约束保证 |
| 并发控制 | 不支持 | 支持事务和锁机制 |
| 安全性 | 仅靠操作系统 | 多层级权限控制 |
| 故障恢复 | 手动 | 自动（日志+备份） |

**核心结论**：文件系统面向单一应用，数据库系统面向多用户、多应用的共享环境。

#### 数据库系统的特点

1. **数据共享**：多个用户、多个应用可以同时访问同一数据
2. **减少冗余**：数据集中存储，避免重复，节省空间
3. **数据独立性**：应用程序与数据的物理存储和逻辑结构解耦
4. **统一管理**：DBMS 提供统一的数据定义、操作、控制接口
5. **安全控制**：用户认证、权限管理、审计日志

---

### 1.2 数据库系统结构

#### 数据库系统组成

```
用户 → 应用程序 → DBMS → 数据库
                ↑
        数据库管理员(DBA)
```

- **用户**：最终使用数据的人，通过应用程序访问数据库
- **应用程序**：业务逻辑层，调用 SQL 与 DBMS 交互
- **DBMS**：核心软件，负责查询解析、优化、执行、事务管理
- **数据库**：数据在磁盘上的物理存储
- **DBA**：负责数据库的安装、配置、备份、性能调优

#### 三级模式结构

数据库系统的核心架构，将数据抽象为三个层次：

```
外模式（External Schema）
   ↓↑ 外模式/模式映像
模式（Conceptual Schema）
   ↓↑ 模式/内模式映像
内模式（Internal Schema）
```

| 层次 | 又名 | 描述 | 示例 |
|------|------|------|------|
| **外模式** | 子模式 / 用户视图 | 用户看到的数据视图，不同用户有不同外模式 | 学生只能看成绩，老师能看全班成绩 |
| **模式** | 逻辑模式 / 概念模式 | 数据库中全部数据的逻辑结构描述 | 学生表、课程表、成绩表的完整定义 |
| **内模式** | 存储模式 | 数据在物理存储介质上的组织方式 | 数据文件存放路径、索引使用 B+ 树存储 |

#### 两级映像与数据独立性

| 独立性类型 | 映像层次 | 含义 | 示例 |
|------------|----------|------|------|
| **逻辑独立性** | 外模式/模式映像 | 模式改变时，修改映像即可，外模式和应用程序无需改变 | 在表中增加一个字段，不影响已有查询视图 |
| **物理独立性** | 模式/内模式映像 | 内模式（存储结构）改变时，修改映像即可，模式和外模式无需改变 | 将数据从一个磁盘迁移到另一个磁盘，应用程序无感知 |

**本质**：三层抽象 + 两级映像，实现了数据与程序的解耦。

## 二、关系模型基础

### 2.1 数据模型概述

数据模型是对现实世界数据特征的抽象，分为三个层次：

| 层次 | 说明 | 代表 |
|------|------|------|
| **概念模型** | 按用户观点建模，与具体 DBMS 无关 | E-R 图 |
| **逻辑模型** | 按计算机观点建模，面向具体 DBMS | 关系模型、层次模型 |
| **物理模型** | 描述数据在存储介质上的组织方式 | 索引结构、存储分配 |

常见逻辑模型：

| 模型 | 数据组织方式 | 现状 |
|------|-------------|------|
| **层次模型** | 树形结构（父-子） | 已淘汰 |
| **网状模型** | 图结构（多对多直接表示） | 基本淘汰 |
| **关系模型** | 二维表 | **当前绝对主流** |

**关系模型由 E.F.Codd 于 1970 年提出**，因其数学基础严谨（关系代数）和结构简单直观而成为主流。

---

### 2.2 关系模型核心概念

#### 关系 = 二维表

一张表就是一个关系（Relation），这是关系模型最基本的概念。

| 术语 | 形式化名称 | 通俗理解 |
|------|-----------|----------|
| **关系** | Relation | 一张二维表 |
| **元组** | Tuple | 表中的一行（一条记录） |
| **属性** | Attribute | 表中的一列（一个字段） |
| **域** | Domain | 属性的取值范围 |
| **分量** | Component | 元组中某个属性的具体值 |

#### 码（Key）

| 概念 | 定义 | 特点 |
|------|------|------|
| **候选码** | 能唯一标识一个元组的属性或属性组 | 可以有多个 |
| **主码** | 从候选码中选定的一个 | 每张表只能有一个 |
| **外码** | 一个表中的属性，是另一张表的主码 | 用于建立表之间的关联 |

> 例：学生表 (学号, 身份证号, 姓名, 班级号)
> - 候选码：学号、身份证号（都能唯一确定一个学生）
> - 主码：选学号作为主码
> - 外码：班级号引用班级表的主码

#### 关系模式 vs 关系实例

- **关系模式**：关系的结构描述（表头），是稳定的。如：`学生(学号, 姓名, 年龄, 班级号)`
- **关系实例**：某一时刻关系中的具体数据（表内容），随时间变化

---

### 2.3 完整性约束

关系数据库通过三类完整性约束保证数据的正确性：

#### 实体完整性

- **规则**：主码的值不能为空（NOT NULL）且必须唯一
- **原因**：主码用于唯一标识一个元组，空值无法标识，重复值会混淆

#### 参照完整性

- **规则**：外码的值要么为空，要么等于被引用表中某个元组的主码值
- **原因**：确保表之间的引用关系有效，不能引用不存在的记录

> 例：学生表的班级号（外码）要么为 NULL，要么必须存在于班级表的班级号中

#### 用户自定义完整性

- **规则**：用户根据业务需求自定义的约束
- 示例：年龄必须在 0~150 之间；性别只能是 "男" 或 "女"；成绩必须在 0~100 之间
- 实现方式：CHECK 约束、NOT NULL、UNIQUE、DEFAULT 等

## 三、SQL 基础与应用

### 3.1 SQL 概述

SQL（Structured Query Language）是关系数据库的标准语言。

#### SQL 的特点

- **综合统一**：集数据定义、操作、查询、控制于一体
- **高度非过程化**：只需说明"做什么"，不必说明"怎么做"
- **面向集合**：操作对象和结果都是元组的集合
- **语言简洁**：核心动词只有 9 个（CREATE、DROP、ALTER、INSERT、UPDATE、DELETE、SELECT、GRANT、REVOKE）

#### SQL 分类

| 类别 | 全称 | 功能 | 核心动词 |
|------|------|------|----------|
| **DDL** | Data Definition Language | 定义数据库对象（库、表、索引等） | CREATE、DROP、ALTER |
| **DML** | Data Manipulation Language | 操作数据（增删改） | INSERT、UPDATE、DELETE |
| **DQL** | Data Query Language | 查询数据 | SELECT |
| **DCL** | Data Control Language | 控制用户权限 | GRANT、REVOKE |

---

### 3.2 数据库与表操作

#### 数据库操作

```sql
-- 创建数据库
CREATE DATABASE student_db;

-- 删除数据库
DROP DATABASE student_db;

-- 使用数据库
USE student_db;
```

#### 表操作

```sql
-- 创建表
CREATE TABLE student (
    id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(50) NOT NULL,
    age INT DEFAULT 18,
    class_id INT,
    FOREIGN KEY (class_id) REFERENCES class(id)
);

-- 修改表：添加字段
ALTER TABLE student ADD COLUMN email VARCHAR(100);

-- 修改表：修改字段类型
ALTER TABLE student MODIFY COLUMN name VARCHAR(100);

-- 修改表：删除字段
ALTER TABLE student DROP COLUMN email;

-- 删除表
DROP TABLE student;
```

#### 常见数据类型

| 类型 | MySQL 示例 | 说明 |
|------|-----------|------|
| 整数 | INT、BIGINT、TINYINT | 分别占 4、8、1 字节 |
| 小数 | DECIMAL(10,2)、FLOAT、DOUBLE | DECIMAL 精确，FLOAT/DOUBLE 近似 |
| 字符串 | CHAR(10) 定长、VARCHAR(255) 变长、TEXT 长文本 | CHAR 固定长度，VARCHAR 可变 |
| 日期时间 | DATE、TIME、DATETIME、TIMESTAMP | DATETIME 范围更大，TIMESTAMP 自动时区转换 |

#### 常见约束

| 约束 | 关键字 | 含义 |
|------|--------|------|
| 主键约束 | PRIMARY KEY | 非空 + 唯一，每表一个 |
| 外键约束 | FOREIGN KEY | 参照另一张表的主键 |
| 非空约束 | NOT NULL | 字段值不能为 NULL |
| 唯一约束 | UNIQUE | 字段值不能重复，但可以为 NULL |
| 默认值 | DEFAULT | 未指定值时使用默认值 |
| 检查约束 | CHECK | 自定义条件，如 CHECK (age >= 0) |

### 3.3 数据增删改（CRUD）

#### INSERT — 插入数据

```sql
-- 单行插入
INSERT INTO student (name, age, class_id) VALUES ('张三', 20, 1);

-- 多行插入
INSERT INTO student (name, age, class_id) VALUES
    ('李四', 21, 1),
    ('王五', 19, 2);

-- 从查询结果插入
INSERT INTO student_backup SELECT * FROM student;
```

#### UPDATE — 修改数据

```sql
-- 修改满足条件的数据
UPDATE student SET age = 21 WHERE name = '张三';

-- 批量修改（慎用！必须加 WHERE）
UPDATE student SET age = age + 1 WHERE class_id = 1;
```

> ⚠️ 不加 WHERE 会修改全表！

#### DELETE — 删除数据

```sql
-- 删除满足条件的数据
DELETE FROM student WHERE name = '张三';

-- 删除全表数据（逐行删除，可回滚）
DELETE FROM student;

-- 清空表（直接删除数据页，不可回滚，更快）
TRUNCATE TABLE student;
```

#### DELETE 与 TRUNCATE 的区别

| 区别 | DELETE | TRUNCATE |
|------|--------|----------|
| 执行方式 | 逐行删除，记录日志 | 删除整个数据页 |
| 速度 | 慢 | 快 |
| 事务回滚 | 可以回滚 | 不可回滚（DDL 语句） |
| WHERE 条件 | 支持 | 不支持 |
| 自增计数器 | 不重置 | 重置 |
| 触发器 | 触发 DELETE 触发器 | 不触发 |

### 3.4 单表查询

#### SELECT 基本语法

```sql
SELECT [DISTINCT] 列名1, 列名2, ...
FROM 表名
[WHERE 条件]
[GROUP BY 分组列]
[HAVING 分组后条件]
[ORDER BY 排序列 ASC|DESC]
[LIMIT 偏移量, 行数];
```

**书写顺序 ≠ 执行顺序**。SQL 的实际执行顺序为：
```
FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT
```

#### WHERE 条件查询

```sql
-- 比较运算符：=, >, <, >=, <=, <> / !=
SELECT * FROM student WHERE age > 20;

-- 逻辑运算符：AND, OR, NOT
SELECT * FROM student WHERE age > 20 AND class_id = 1;

-- BETWEEN（闭区间）
SELECT * FROM student WHERE age BETWEEN 18 AND 22;

-- IN（多值匹配）
SELECT * FROM student WHERE class_id IN (1, 2, 3);

-- LIKE（模糊匹配）
-- % 匹配任意多个字符，_ 匹配任意单个字符
SELECT * FROM student WHERE name LIKE '张%';   -- 姓张的
SELECT * FROM student WHERE name LIKE '张_';   -- 姓张且两字

-- IS NULL / IS NOT NULL
SELECT * FROM student WHERE email IS NULL;
```

#### ORDER BY 排序

```sql
-- 升序（默认）
SELECT * FROM student ORDER BY age ASC;

-- 降序
SELECT * FROM student ORDER BY age DESC;

-- 多字段排序
SELECT * FROM student ORDER BY class_id ASC, age DESC;
```

#### LIMIT 分页

```sql
-- 取前 10 条
SELECT * FROM student LIMIT 10;

-- 跳过 10 条后取 10 条（第 2 页）
SELECT * FROM student LIMIT 10, 10;

-- 通用分页公式：LIMIT (page-1) * pageSize, pageSize
-- 第 3 页，每页 10 条：
SELECT * FROM student LIMIT 20, 10;
```

#### DISTINCT 去重

```sql
SELECT DISTINCT class_id FROM student;
```

- DISTINCT 作用于所有选择的列的组合

### 3.5 聚合查询

#### 聚合函数

| 函数 | 作用 | 注意 |
|------|------|------|
| COUNT(\*) | 统计总行数（含 NULL） | 最常用 |
| COUNT(列名) | 统计该列非 NULL 的行数 | 不含 NULL |
| SUM(列名) | 求和 | 忽略 NULL |
| AVG(列名) | 求平均值 | 忽略 NULL |
| MAX(列名) | 求最大值 | 忽略 NULL |
| MIN(列名) | 求最小值 | 忽略 NULL |

```sql
-- 统计学生总数
SELECT COUNT(*) FROM student;

-- 统计平均年龄
SELECT AVG(age) FROM student;
```

#### GROUP BY 分组

```sql
-- 按班级统计人数
SELECT class_id, COUNT(*) AS student_count
FROM student
GROUP BY class_id;
```

> 规则：SELECT 中出现的非聚合列，必须出现在 GROUP BY 中。

#### HAVING 分组后筛选

```sql
-- 查询人数大于 30 的班级
SELECT class_id, COUNT(*) AS cnt
FROM student
GROUP BY class_id
HAVING cnt > 30;
```

#### WHERE 与 HAVING 的区别

| 区别 | WHERE | HAVING |
|------|-------|--------|
| 作用阶段 | 分组前筛选（FROM 之后） | 分组后筛选（GROUP BY 之后） |
| 能否用聚合函数 | 不能 | 能（HAVING COUNT(*) > 10） |
| 执行效率 | 先过滤，减少分组数据量 | 分组完成后再过滤 |

```sql
-- WHERE + HAVING 同时使用
SELECT class_id, AVG(age) AS avg_age
FROM student
WHERE age >= 18          -- 先筛掉未满 18 的
GROUP BY class_id
HAVING avg_age > 20;     -- 再筛掉平均年龄不足 20 的班级
```

### 3.6 多表查询

#### 内连接 INNER JOIN

返回两表中匹配的行。

```sql
-- 隐式内连接
SELECT s.name, c.class_name
FROM student s, class c
WHERE s.class_id = c.id;

-- 显式内连接（推荐）
SELECT s.name, c.class_name
FROM student s
INNER JOIN class c ON s.class_id = c.id;
```

#### 外连接

| 连接类型 | 关键字 | 结果 |
|----------|--------|------|
| 左外连接 | LEFT JOIN | 左表全部 + 右表匹配的，右表无匹配则填 NULL |
| 右外连接 | RIGHT JOIN | 右表全部 + 左表匹配的，左表无匹配则填 NULL |

```sql
-- 左外连接：查询所有学生及其班级（即使没分班）
SELECT s.name, c.class_name
FROM student s
LEFT JOIN class c ON s.class_id = c.id;
```

#### 子查询

一个查询嵌套在另一个查询中。

```sql
-- 查询比平均年龄大的学生
SELECT * FROM student
WHERE age > (SELECT AVG(age) FROM student);

-- 查询有学生的班级信息
SELECT * FROM class
WHERE id IN (SELECT DISTINCT class_id FROM student WHERE class_id IS NOT NULL);
```

#### EXISTS 与 IN 的区别

| 区别 | IN | EXISTS |
|------|-----|--------|
| 原理 | 先执行子查询，再与外表匹配 | 外表每行代入子查询判断是否存在 |
| 子查询结果大 | 性能差（结果集大） | 性能好（找到即停止） |
| NULL 处理 | 子查询含 NULL 时可能返回未知 | 不受 NULL 影响 |

```sql
-- 用 EXISTS 判断是否有学生
SELECT * FROM class c
WHERE EXISTS (
    SELECT 1 FROM student s WHERE s.class_id = c.id
);
```

> **建议**：子查询结果集小时用 IN，大时用 EXISTS。多数情况下 EXISTS 更稳定。

## 四、E-R 模型与表设计入门

### 4.1 数据库设计流程

```
需求分析 → 概念结构设计 → 逻辑结构设计 → 物理结构设计 → 数据库实施与维护
```

| 阶段 | 工作内容 | 产出物 |
|------|---------|--------|
| **需求分析** | 了解业务流程、数据需求、约束条件 | 需求规格说明书 |
| **概念设计** | 抽象出实体、属性、联系 | E-R 图 |
| **逻辑设计** | E-R 图转换为关系模式 | 关系模式集合（表结构） |
| **物理设计** | 选择存储结构、索引策略 | 物理数据库 |
| **实施维护** | 建库建表、数据迁移、性能优化 | 运行中的数据库 |

---

### 4.2 E-R 模型

#### 基本元素

| 元素 | E-R 图符号 | 说明 |
|------|-----------|------|
| **实体** | 矩形 | 客观存在且可相互区分的事物，如学生、课程 |
| **属性** | 椭圆 | 实体的特征，如学号、姓名 |
| **联系** | 菱形 | 实体之间的关联 |

#### 联系的类型（映射基数）

| 类型 | 例 | 说明 |
|------|-----|------|
| **一对一（1:1）** | 学生 ↔ 校园卡 | 一个学生有一张卡，一张卡属于一个学生 |
| **一对多（1:N）** | 班级 ↔ 学生 | 一个班级有多个学生，一个学生属于一个班级 |
| **多对多（M:N）** | 学生 ↔ 课程 | 一个学生选多门课，一门课有多个学生 |

#### E-R 图向关系模式转换

- **实体 → 表**：每个实体转换为一张表，属性变为字段
- **1:1 联系**：可合并到任一端，或在任一端添加外键
- **1:N 联系**：在 N 端添加外键，指向 1 端的主键
- **M:N 联系**：必须单独创建中间表（关联表），包含两端主键作为外键

> 例：学生选课（M:N）→ 选课表(学号, 课程号, 成绩)，学号和课程号联合作为主键

---

### 4.3 表设计基本准则

#### 字段命名规范

- 使用小写字母 + 下划线（snake_case）：`user_name`、`created_at`
- 字段名见名知意，避免缩写：`create_time` 比 `ct` 好
- 布尔字段用 `is_` 前缀：`is_deleted`、`is_active`
- 主键统一用 `id`，关联外键用 `关联表_id`：`user_id`

#### 主键设计原则

- 使用自增整型或有序 UUID（如雪花算法），避免随机 UUID（导致页分裂）
- 不建议使用业务字段作为主键（业务可能变化）
- 强烈建议每张表都有主键

#### 字段类型选择原则

- **年龄**：TINYINT UNSIGNED（0~255 足够）
- **金额**：DECIMAL（精确），绝不用 FLOAT/DOUBLE
- **状态**：TINYINT + 注释说明（而非直接用字符串）
- **长文本**：TEXT 而非过大的 VARCHAR
- **时间**：DATETIME 或 TIMESTAMP（根据需求）

#### 外键与关联设计

- 外键列名通常为 `关联表名_id`
- 多对多关系需要中间表，中间表通常以两端 ID 联合作为主键
- 建立外键约束有助于保证数据完整性，但高并发场景下可能选择应用层维护

## 五、索引基础

### 5.1 什么是索引

> **索引**是数据库表中一列或多列的值进行排序后建立的数据结构，用于加速查询。

**类比**：书的目录——要找某个章节，翻目录比逐页翻快得多。

#### 索引的作用

- 大幅提升查询速度（从全表扫描变为索引查找）
- 加速 ORDER BY 排序和 GROUP BY 分组
- 保证数据唯一性（唯一索引）

#### 索引的代价

- 占用额外的磁盘空间
- 降低写入速度（INSERT、UPDATE、DELETE 时需同时更新索引）
- 过多的索引可能导致优化器选错索引

---

### 5.2 常见索引类型

#### 按功能分类

| 索引类型 | 特点 | 每表数量 |
|----------|------|----------|
| **主键索引** | 唯一且非空，自动创建 | 1 个 |
| **唯一索引** | 值唯一，允许 NULL | 多个 |
| **普通索引** | 仅加速查询，无约束 | 多个 |
| **联合索引** | 多列组合为一个索引 | 多个 |

```sql
-- 创建普通索引
CREATE INDEX idx_name ON student(name);

-- 创建唯一索引
CREATE UNIQUE INDEX idx_email ON student(email);

-- 创建联合索引
CREATE INDEX idx_name_age ON student(name, age);
```

#### 聚簇索引 vs 非聚簇索引

| 区别 | 聚簇索引 | 非聚簇索引 |
|------|----------|------------|
| 数据存储 | 叶子节点存储完整行数据 | 叶子节点存储主键值 |
| 每表数量 | 只能有 1 个 | 可以有多个 |
| 物理顺序 | 数据行按索引顺序物理存储 | 索引结构与数据分开存储 |
| 查找速度 | 找到索引即找到数据 | 需要回表（除非覆盖索引） |

> InnoDB 的主键索引是聚簇索引，数据即索引，索引即数据。

#### 回表查询与覆盖索引

- **回表查询**：通过非聚簇索引查到的只是主键值，还需要用主键去聚簇索引中查完整数据（多一次查询）
- **覆盖索引**：查询所需的列都在索引中，无需回表，效率最高

```sql
-- 假设建立了 idx_name_age(name, age) 索引
-- 覆盖索引（不用回表）：SELECT 的列都在索引中
SELECT name, age FROM student WHERE name = '张三';

-- 需要回表：SELECT 了不在索引中的列
SELECT name, age, email FROM student WHERE name = '张三';
```

---

### 5.3 索引使用原则

#### 最左前缀原则

联合索引 `(a, b, c)` 等效于创建了 `(a)`、`(a, b)`、`(a, b, c)` 三个索引。

```sql
-- 假设索引 idx_test(a, b, c)
-- 走索引：
WHERE a = 1           -- 走 a
WHERE a = 1 AND b = 2 -- 走 a, b
WHERE a = 1 AND c = 3 -- 只走 a（b 断了）

-- 不走索引：
WHERE b = 2           -- 缺少最左列 a
WHERE c = 3           -- 缺少最左列 a
```

#### 适合与不适合建索引的场景

| 适合建索引 | 不适合建索引 |
|-----------|-------------|
| 频繁作为 WHERE 查询条件的字段 | 频繁更新的字段（维护代价大） |
| 频繁 ORDER BY 排序的字段 | 数据量很小的表（全表扫描更快） |
| 频繁 GROUP BY 分组的字段 | 区分度很低的字段（如性别） |
| 表关联时的外键列 | - |

#### EXPLAIN 分析执行计划

```sql
EXPLAIN SELECT * FROM student WHERE name = '张三';
```

关注字段：
| 字段 | 含义 | 理想值 |
|------|------|--------|
| type | 访问类型 | const > ref > range > index > ALL（全表扫描最差） |
| key | 使用的索引 | 期望不为 NULL |
| rows | 扫描的行数 | 越小越好 |
| Extra | 额外信息 | Using index（覆盖索引）最好 |

## 六、事务基础

### 6.1 事务概念

> **事务**是数据库操作的逻辑单元，这组操作要么全部成功执行，要么全部不执行。

**经典示例——银行转账**：A 向 B 转账 100 元：
1. A 账户减 100
2. B 账户加 100

这两步必须要么都执行，要么都不执行。不能出现 A 减了钱但 B 没收到的情况。

#### 基本操作

```sql
BEGIN;       -- 开始事务
-- 一组 SQL 操作
COMMIT;      -- 提交，确认所有修改
-- 或
ROLLBACK;    -- 回滚，撤销所有修改
```

> MySQL 中默认每条语句自动提交。需显式 BEGIN 开启手动事务。

---

### 6.2 ACID 特性

| 特性 | 全称 | 含义 |
|------|------|------|
| **A** | Atomicity 原子性 | 事务中的操作不可分割，要么全做，要么全不做 |
| **C** | Consistency 一致性 | 事务执行前后，数据库从一种一致状态到另一种一致状态 |
| **I** | Isolation 隔离性 | 并发执行的事务互不干扰 |
| **D** | Durability 持久性 | 事务一旦提交，数据永久保存，即使系统崩溃 |

**四者关系**：
- 原子性是基础，保证要么全做要么全不做
- 一致性是目标，事务前后的数据都满足约束
- 隔离性是并发场景下保证一致性的手段
- 持久性是最终保障，提交后数据不丢失

---

### 6.3 并发问题与隔离级别

#### 并发问题

| 问题 | 说明 | 场景 |
|------|------|------|
| **脏读** | 读到了其他事务未提交的数据 | B 读到 A 修改但未提交的数据，A 回滚后 B 读到的是脏数据 |
| **不可重复读** | 同一事务内两次读取同一数据结果不同 | A 读了一条数据，B 修改并提交，A 再读发现数据变了 |
| **幻读** | 同一事务内两次查询返回的行数不同 | A 查了 10 条，B 插入一条并提交，A 再查变成 11 条 |
| **丢失更新** | 后提交的事务覆盖了先提交事务的修改 | A 和 B 同时修改同一行，后提交的覆盖先提交的 |

#### 四种隔离级别

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 并发性能 |
|----------|------|-----------|------|----------|
| **READ UNCOMMITTED** | ✗ | ✗ | ✗ | 最高 |
| **READ COMMITTED** | ✓ | ✗ | ✗ | 较高 |
| **REPEATABLE READ** | ✓ | ✓ | ✗ | 中等 |
| **SERIALIZABLE** | ✓ | ✓ | ✓ | 最低 |

- ✓ = 已解决，✗ = 可能发生
- MySQL InnoDB 默认隔离级别为 **REPEATABLE READ**

```sql
-- 查看当前隔离级别
SELECT @@transaction_isolation;

-- 设置隔离级别
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;
```

### 7.2 下节预告：规范化理论与工程实践

#### 规范化理论——为什么"凭直觉建表"会出问题？

我们会用一张"大表"演示，逐步拆解出三个问题：

1. **数据冗余**：同一个信息存了 N 遍
2. **插入异常**：想加新班级却因为没有学生而加不进去
3. **删除/更新异常**：删掉最后一个学生，班级信息也丢了

然后引出**三大范式**，一步步把表拆好：

| 范式 | 一句话 | 解决的问题 |
|------|--------|-----------|
| 1NF | 每列不可再分 | 表结构规范化 |
| 2NF | 消除部分依赖 | 非主属性必须完全依赖主键 |
| 3NF | 消除传递依赖 | 非主属性不能通过另一个非主属性依赖主键 |

还会讲到什么时候该"违反"范式（反规范化），以及：

- SQL 注入防护与参数化查询
- 数据库连接池
- 慢查询排查
- 读写分离、分库分表、缓存策略等进阶方向
