# 浏览器核心原理与技术指南

浏览器渲染流程、缓存机制、跨域处理、本地存储等前端基础原理。

---

## 一、浏览器渲染流程

### 1.1 从输入 URL 到页面展示

```
DNS 解析 → TCP 连接 → TLS 握手(HTTPS) → HTTP 请求 → 服务器响应 → 解析渲染
```

### 1.2 渲染引擎的完整流水线

```
HTML ──→ DOM Tree ──────────────┐
                                ├──→ Render Tree ──→ Layout ──→ Paint ──→ Composite
CSS  ──→ CSSOM Tree ────────────┘
```

| 阶段 | 产物 | 说明 |
|---|---|---|
| **Parse** | DOM / CSSOM | 字节 → 字符 → Token → Node → Tree |
| **Style** | Render Tree | 合并 DOM + CSSOM，剔除 `display:none` 节点 |
| **Layout** | Layout Tree | 计算每个节点的几何位置与尺寸 |
| **Paint** | Paint Records | 将 Layout 结果拆解为绘制指令 |
| **Composite** | Layer | 分图层光栅化并合成输出到屏幕 |

### 1.3 关键渲染路径（Critical Rendering Path）

**阻塞渲染的资源：**

```html
<!-- 1. CSS 阻塞渲染（但不会阻塞 HTML 解析） -->
<link rel="stylesheet" href="style.css">

<!-- 2. 同步 JS 阻塞解析（遇到即暂停 HTML 解析，等待下载执行） -->
<script src="app.js"></script>

<!-- 3. async —— 异步下载，下载完立即执行（阻塞解析），执行顺序不定 -->
<script async src="analytics.js"></script>

<!-- 4. defer —— 异步下载，DOM 解析完成后、DOMContentLoaded 前按序执行 -->
<script defer src="framework.js"></script>
```

| 属性 | 下载 | 执行时机 | 执行顺序 |
|---|---|---|---|
| 无 | 阻塞解析 | 立即 | 文档序 |
| `async` | 并行 | 下载完立即执行 | 谁先下载完谁先执行 |
| `defer` | 并行 | DOM 解析完成后 | 文档序 |

**首屏优化策略**：
- 关键 CSS 内联到 `<head>`，非关键 CSS 延迟加载
- JS 放在 `</body>` 前或加 `defer`，避免阻塞解析
- 减少关键渲染路径的资源数量与字节大小

### 1.4 重排（Reflow）与重绘（Repaint）

```
重排必定触发重绘，重绘不一定触发重排。
```

| 操作 | 触发条件 | 代价 |
|---|---|---|
| **重排** | 修改几何属性（width/height/padding/top）、DOM 增删、窗口变化、读取 `offsetWidth`/`scrollTop` 等 | 高 |
| **重绘** | 修改外观属性（color/background/visibility/box-shadow） | 中 |
| **合成** | 仅 `transform` / `opacity` | 低（仅 GPU 合成，不触动 Layout/Paint） |

**强制同步布局（Layout Thrashing）**：

```javascript
// ❌ 经典反模式：反复读 offsetHeight 写 height，触发多次重排
elements.forEach(el => {
    const h = el.offsetHeight;  // 读取，强制同步布局
    el.style.height = h + 10 + 'px';  // 写入，使布局失效
    // 下一轮读又触发一次新的布局…
});

// ✅ requestAnimationFrame 批量处理
requestAnimationFrame(() => {
    elements.forEach(el => {
        const h = el.offsetHeight;
        el.style.height = h + 10 + 'px';
    });
});
```

**减少重排的实践**：
- 批量修改 DOM：用 `DocumentFragment` 或先 `display:none` → 修改 → 恢复显示
- 避免在 `for` 循环中使用 `el.style.xxx = yyy`，改用 classList 切换
- 动画使用 `transform` 和 `opacity`（跳过 Layout 阶段）
- 对频繁读取的布局属性进行缓存

### 1.5 图层合成

浏览器将页面分为多个图层独立绘制，最终合成到一起。

**创建独立图层的条件**：
- 根节点、`<video>`、`<canvas>`、`<iframe>`
- `will-change: transform` 明确告知浏览器
- 3D transform（`translateZ(0)` 是经典的强制 GPU 加速 hack）
- `position: fixed` / `position: sticky`

**注意**：不要滥用图层（`will-change: transform` 到处加），过多图层会消耗 GPU 内存。

## 二、HTTP 缓存机制

### 2.1 缓存体系总览

```
浏览器缓存
├── 强缓存（直接使用，不发请求）
│   ├── Expires (HTTP/1.0)
│   └── Cache-Control (HTTP/1.1)  优先级 > Expires
│
└── 协商缓存（发请求验证，304 再使用）
    ├── Last-Modified / If-Modified-Since
    └── ETag / If-None-Match        优先级 > Last-Modified
```

### 2.2 强缓存

命中强缓存时不发送请求，状态码显示 `200 (from disk cache)` 或 `200 (from memory cache)`。

```http
Cache-Control: max-age=3600         # 缓存 3600 秒（相对时间）
Cache-Control: no-cache             # 可缓存，但每次必须验证（走协商缓存）
Cache-Control: no-store             # 禁止缓存
Cache-Control: public               # 任何中间节点（CDN）都可缓存
Cache-Control: private              # 仅浏览器可缓存
Cache-Control: immutable            # 资源内容永不变，强制缓存期内不验证

Expires: Thu, 01 Dec 2026 16:00:00 GMT  # 绝对时间，受客户端时钟影响
```

**`memory cache` vs `disk cache`**：
- memory cache：当前 tab 生命周期，速度快，容量小
- disk cache：跨 tab、跨会话，速度慢，容量大
- 典型模式：base64 小图、已解析的 JS 进 memory；大文件、CSS 进 disk

### 2.3 协商缓存

强缓存过期后进入协商缓存阶段：

```http
# 请求头
If-Modified-Since: Wed, 21 Jun 2026 10:00:00 GMT   # 与 Last-Modified 配对
If-None-Match: "abc123"                            # 与 ETag 配对

# 响应头（200 时完整返回；304 时仅返回头、无 body）
Last-Modified: Wed, 21 Jun 2026 10:00:00 GMT
ETag: "abc123"
```

**ETag 优先于 Last-Modified**：
- Last-Modified 精度为秒，1 秒内多次变更会丢失
- 文件内容不变但 mtime 变化（如打包部署）时，Last-Modified 误判变更
- ETag 是内容哈希，精确反映内容变化

### 2.4 缓存决策流程

```
发起请求
  │
  ├── Cache-Control: no-store?
  │     └─ YES → 直接请求服务器
  │
  ├── 强缓存未过期？
  │     └─ YES → 200 from cache（不发请求）
  │
  ├── 有 ETag？
  │     ├─ YES → 发 If-None-Match
  │     └─ NO  → 发 If-Modified-Since
  │
  ├── 服务器判断资源未变？
  │     ├─ YES → 304 Not Modified（用缓存）
  │     └─ NO  → 200 OK（返回新资源）
```

### 2.5 实战缓存策略

| 资源类型 | Cache-Control | 更新机制 |
|---|---|---|
| HTML 入口 | `no-cache` | 每次验证，确保拿到最新 JS/CSS 引用 |
| JS / CSS（带 hash） | `max-age=31536000, immutable` | 文件名 hash 变化 = 新 URL，永久缓存 |
| 图片 / 字体 | `max-age=2592000` (30 天) | 配合 hash 命名 |
| API 数据 | `no-store` 或短 `max-age` | 按业务需要 |

```
构建产物的典型命名：app.a8f3b2c.js  → 内容变则文件名变
```

## 三、跨域处理（CORS 与代理）

### 3.1 同源策略

**源（Origin）= 协议 + 域名 + 端口**，三者任一不同即为跨域。

| 场景 | 是否跨域 | 原因 |
|---|---|---|
| `http://a.com` → `https://a.com` | 是 | 协议不同 |
| `http://a.com` → `http://b.a.com` | 是 | 子域名不同 |
| `http://a.com:80` → `http://a.com:8080` | 是 | 端口不同 |
| `http://a.com/a` → `http://a.com/b` | 否 | 同源 |

**同源策略限制**：
- DOM 访问（iframe 间）
- Cookie / LocalStorage / IndexedDB 读取
- AJAX 请求（`fetch`、`XMLHttpRequest`）

**不受同源策略限制**：
- `<script>` 标签（JSONP 的基础）
- `<img>` / `<link>` / `<video>` / `<audio>`
- `<iframe>` 嵌入页面本身（但不允许脚本互操作）

### 3.2 CORS（跨域资源共享）

**简单请求**（同时满足所有条件）：
- 方法为 `GET` / `HEAD` / `POST`
- 仅使用 Accept、Accept-Language、Content-Language、Content-Type
- Content-Type 仅限 `text/plain`、`multipart/form-data`、`application/x-www-form-urlencoded`

```http
# 请求
GET /api/data HTTP/1.1
Origin: http://localhost:3000

# 响应
Access-Control-Allow-Origin: http://localhost:3000
# 或
Access-Control-Allow-Origin: *    # 注意：* 无法携带 Cookie
```

**非简单请求**（预检请求 Preflight）：

```http
# 预检请求 —— OPTIONS
OPTIONS /api/data HTTP/1.1
Origin: http://localhost:3000
Access-Control-Request-Method: PUT
Access-Control-Request-Headers: X-Custom-Header, Content-Type

# 预检响应
Access-Control-Allow-Origin: http://localhost:3000
Access-Control-Allow-Methods: GET, POST, PUT, DELETE
Access-Control-Allow-Headers: X-Custom-Header, Content-Type
Access-Control-Max-Age: 86400       # 预检缓存秒数

# 预检通过后，发出真正请求
PUT /api/data HTTP/1.1
Origin: http://localhost:3000
Access-Control-Allow-Origin: http://localhost:3000
```

**携带 Cookie**：

```javascript
// 前端
fetch('/api/user', { credentials: 'include' });

// 或 XHR
xhr.withCredentials = true;
```

```http
# 服务端响应头
Access-Control-Allow-Origin: http://localhost:3000   # 必须是具体域名，不能是 *
Access-Control-Allow-Credentials: true
```

### 3.3 跨域解决方案对比

| 方案 | 原理 | 适用场景 | 改动方 |
|---|---|---|---|
| **CORS** | 服务端设置响应头 | 可控的后端 API | 后端 |
| **代理（Proxy）** | 同源服务器转发请求 | 开发环境（devServer.proxy） | 前端/运维 |
| **JSONP** | `<script>` 不受同源限制，仅 GET | 老旧接口兼容 | 前后端 |
| **postMessage** | `window.postMessage` 安全跨文档通信 | iframe 间、窗口间通信 | 前后端 |
| **Nginx 反向代理** | 转发到同域下的不同路径 | 生产环境统一入口 | 运维 |
| **WebSocket** | 不受同源策略限制 | 实时双向通信 | 前后端 |

### 3.4 Nginx 反向代理示例

```nginx
location /api/ {
    proxy_pass http://backend-server:8080/;

    # 隐藏后端真正地址
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
}
```

前端请求 `http://frontend.com/api/data`，Nginx 转发到 `http://backend-server:8080/data`，浏览器感知不到跨域。

### 3.5 postMessage 安全跨文档通信

```javascript
// 父页面
iframe.contentWindow.postMessage(
    { type: 'greet', payload: 'hello' },
    'https://trusted-child.com'  // 明确指定目标 origin，不要用 *
);

// 子页面接收
window.addEventListener('message', (event) => {
    // 必须验证 origin
    if (event.origin !== 'https://trusted-parent.com') return;
    console.log(event.data);
});

// 子页面回复
event.source.postMessage({ type: 'reply' }, event.origin);
```

## 四、本地存储

### 4.1 四种存储方案对比

| 特性 | Cookie | localStorage | sessionStorage | IndexedDB |
|---|---|---|---|---|
| 容量 | 4KB | 5MB~10MB | 5MB~10MB | 无上限（需授权） |
| 生命周期 | 可设过期时间 | 永久（手动清除） | 标签页关闭即清 | 永久 |
| 作用域 | 同源 + path | 同源 | 同源 + 同标签页 | 同源 |
| 随请求发送 | 是（自动附带） | 否 | 否 | 否 |
| API | `document.cookie` | 同步 API | 同步 API | 异步 API |
| 数据类型 | 字符串 | 字符串 | 字符串 | 任意（结构化克隆） |

### 4.2 Cookie 深度解析

```javascript
// 设置 Cookie
document.cookie = 'token=abc123; max-age=3600; path=/; domain=.example.com; secure; samesite=lax';

// 读取（只能拿到键值，拿不到属性）
console.log(document.cookie);  // "token=abc123; theme=dark"
```

**Cookie 属性说明**：

| 属性 | 作用 |
|---|---|
| `max-age` | 存活秒数，负数 = 立即删除，0 = 删除 |
| `expires` | GMT 绝对过期时间 |
| `domain` | 作用域名；`.example.com` 覆盖所有子域 |
| `path` | 作用路径；`/` 全站生效 |
| `secure` | 仅 HTTPS 传输 |
| `samesite` | Strict / Lax / None —— CSRF 防御关键 |
| `httpOnly` | 禁止 JS 访问（仅服务端可设），防 XSS 窃取 |

**SameSite 详解**：

| 值 | 行为 |
|---|---|
| `Strict` | 跨站请求不携带 Cookie（点击第三方链接跳转也不带） |
| `Lax` | 仅顶级导航（`<a>` 点击、`<link>`、GET 表单）携带；iframe/img/ajax 不携带（默认值） |
| `None` | 跨站请求全部携带，**必须同时设置 `Secure`** |

```http
Set-Cookie: session_id=abc; Secure; HttpOnly; SameSite=Lax
```

### 4.3 Web Storage 与存储事件

```javascript
// localStorage / sessionStorage 共享相同 API
localStorage.setItem('key', 'value');
const val = localStorage.getItem('key');
localStorage.removeItem('key');
localStorage.clear();

// 对象需 JSON 序列化
localStorage.setItem('user', JSON.stringify({ name: 'Alice' }));
const user = JSON.parse(localStorage.getItem('user'));

// storage 事件 —— 仅在不同页面间触发，同页面不会触发
window.addEventListener('storage', (e) => {
    console.log(e.key);       // 变化的 key
    console.log(e.oldValue);  // 旧值
    console.log(e.newValue);  // 新值
    console.log(e.url);       // 触发变动的页面 URL
});
```

### 4.4 IndexedDB 基础操作

适合大批量结构化数据（Blob、File、对象）。

```javascript
// 打开数据库
const request = indexedDB.open('MyDB', 1);

// 版本升级时创建对象仓库与索引
request.onupgradeneeded = (event) => {
    const db = event.target.result;
    const store = db.createObjectStore('users', { keyPath: 'id' });
    store.createIndex('name_idx', 'name', { unique: false });
};

request.onsuccess = (event) => {
    const db = event.target.result;

    // 写
    const tx = db.transaction('users', 'readwrite');
    tx.objectStore('users').put({ id: 1, name: 'Alice', age: 25 });

    // 读
    const getReq = tx.objectStore('users').get(1);
    getReq.onsuccess = () => console.log(getReq.result);
};

// Promise 化封装推荐使用 idb 库（https://github.com/jakearchibald/idb）
```

## 五、网络请求与数据通信

### 5.1 XMLHttpRequest vs fetch

```javascript
// XHR（旧）
const xhr = new XMLHttpRequest();
xhr.open('POST', '/api/data');
xhr.setRequestHeader('Content-Type', 'application/json');
xhr.onload = () => {
    if (xhr.status >= 200 && xhr.status < 300) {
        JSON.parse(xhr.responseText);
    }
};
xhr.onerror = () => console.error('network error');
xhr.send(JSON.stringify({ key: 'value' }));

// fetch（新）
const response = await fetch('/api/data', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ key: 'value' }),
    credentials: 'include',  // 携带 Cookie
});

if (!response.ok) throw new Error(`HTTP ${response.status}`);
const data = await response.json();
```

**fetch vs XHR 差异**：

| 特性 | XHR | fetch |
|---|---|---|
| Promise | 不支持（需封装） | 原生支持 |
| 请求取消 | `xhr.abort()` | `AbortController` |
| 下载进度 | `onprogress` | 不支持（需 `ReadableStream`） |
| 超时 | `timeout` 属性 | 需 `AbortController` + `setTimeout` |
| HTTP 错误码 | 仅 onload | 不 reject（需手动检查 `response.ok`） |

### 5.2 AbortController

```javascript
const controller = new AbortController();

fetch('/api/data', { signal: controller.signal })
    .then(res => res.json())
    .catch(err => {
        if (err.name === 'AbortError') {
            console.log('request cancelled');
        }
    });

// 取消请求
controller.abort();

// 超时控制
setTimeout(() => controller.abort(), 5000);

// 一个 controller 可取消多个请求（signal 可共享）
fetch('/api/a', { signal: controller.signal });
fetch('/api/b', { signal: controller.signal });
controller.abort();  // 两个请求都取消
```

### 5.3 Server-Sent Events (SSE)

单向推送：服务端 → 客户端。自动重连，基于 HTTP，比 WebSocket 简单。

```javascript
const es = new EventSource('/api/stream');

es.addEventListener('message', (e) => {
    console.log(JSON.parse(e.data));
});

// 自定义事件
es.addEventListener('custom-event', (e) => {
    console.log(e.data);
});

es.onerror = () => {
    // EventSource 自动尝试重连
    if (es.readyState === EventSource.CLOSED) {
        console.log('stream closed permanently');
    }
};
```

```http
# 服务端响应头
Content-Type: text/event-stream
Cache-Control: no-cache
Connection: keep-alive

# 数据格式
data: {"message": "hello"}

event: custom-event
data: {"type": "custom"}

# 重连间隔建议
retry: 3000
```

### 5.4 WebSocket

全双工通信，适合聊天、协作编辑、实时仪表盘。

```javascript
const ws = new WebSocket('wss://example.com/ws');

ws.onopen = () => {
    ws.send(JSON.stringify({ type: 'join', room: 'general' }));
};

ws.onmessage = (event) => {
    const msg = JSON.parse(event.data);
    console.log(msg);
};

ws.onclose = (event) => {
    console.log(`closed: code=${event.code}, reason=${event.reason}`);
};

ws.onerror = (err) => {
    console.error('websocket error', err);
};

// 心跳保活（WebSocket nginx/proxy 默认超时通常 60s）
setInterval(() => {
    if (ws.readyState === WebSocket.OPEN) {
        ws.send('ping');  // 或 ws.send(JSON.stringify({ type: 'ping' }))
    }
}, 30000);
```

## 六、文档与资源生命周期

### 6.1 DOMContentLoaded vs load

```javascript
// DOM 解析完成，defer 脚本已执行，但图片/样式/iframe 等外部资源可能未加载完
document.addEventListener('DOMContentLoaded', () => {
    console.log('DOM ready');
});

// 所有资源（图片、样式、iframe）加载完毕
window.addEventListener('load', () => {
    console.log('everything loaded');
});

// 页面即将关闭
window.addEventListener('beforeunload', (e) => {
    e.preventDefault();
    e.returnValue = '';  // 触发确认对话框
});

// 页面不可见时暂停非必要任务
document.addEventListener('visibilitychange', () => {
    if (document.hidden) {
        // 暂停轮询、动画
    } else {
        // 恢复
    }
});
```

### 6.2 资源优先级

```html
<!-- preload：声明式 fetch，告诉浏览器"这份资源现在就下载，很重要" -->
<link rel="preload" href="critical-font.woff2" as="font" type="font/woff2" crossorigin>
<link rel="preload" href="hero-image.jpg" as="image">

<!-- prefetch：空闲时下载"未来页面可能用到"的资源 -->
<link rel="prefetch" href="next-page.js" as="script">

<!-- dns-prefetch：提前 DNS 解析 -->
<link rel="dns-prefetch" href="//api.example.com">

<!-- preconnect：提前建立 TCP + TLS 连接 -->
<link rel="preconnect" href="https://api.example.com">

<!-- prerender：预渲染整个页面（谨慎使用） -->
<link rel="prerender" href="/next-page.html">
```

| 策略 | 时机 | 用途 |
|---|---|---|
| `preload` | 当前页面立即 | 关键字体、首图、关键 CSS/JS |
| `prefetch` | 空闲时 | 下一个导航页面资源 |
| `dns-prefetch` | 尽可能早 | 跨域域名 |
| `preconnect` | 尽可能早 | 确定将使用的重要跨域源 |
| `prerender` | 空闲时 | 用户很可能访问的下一页 |

### 6.3 性能指标

| 指标 | 含义 |
|---|---|
| **FP** (First Paint) | 首次绘制任意像素 |
| **FCP** (First Contentful Paint) | 首次绘制内容（文本/图片/非空 canvas） |
| **LCP** (Largest Contentful Paint) | 最大可见内容元素渲染完成 |
| **FID** (First Input Delay) | 首次交互延迟 |
| **INP** (Interaction to Next Paint) | 交互到下次绘制的延迟（替代 FID） |
| **CLS** (Cumulative Layout Shift) | 累计布局偏移 |
| **TTI** (Time to Interactive) | 页面完全可交互 |
| **TBT** (Total Blocking Time) | FCP 到 TTI 之间的长任务阻塞总时长 |

```javascript
// 实战：监听 LCP
new PerformanceObserver((list) => {
    const entries = list.getEntries();
    const last = entries[entries.length - 1];
    console.log('LCP:', last.startTime);
}).observe({ type: 'largest-contentful-paint', buffered: true });

// 监听 CLS
let clsValue = 0;
new PerformanceObserver((list) => {
    for (const entry of list.getEntries()) {
        if (!entry.hadRecentInput) {  // 排除用户输入后的偏移
            clsValue += entry.value;
        }
    }
}).observe({ type: 'layout-shift', buffered: true });
```

## 七、安全基础

### 7.1 XSS（跨站脚本攻击）

| 类型 | 说明 |
|---|---|
| 存储型 | 恶意脚本存入数据库，所有访问用户受影响 |
| 反射型 | 恶意脚本藏在 URL 参数，服务端回显到页面 |
| DOM 型 | 纯前端：`location.hash` 等来源被写入 `innerHTML` |

```javascript
// ❌ 危险
element.innerHTML = userInput;

// ✅ 安全
element.textContent = userInput;                 // 纯文本
element.setAttribute('href', sanitize(url));      // 属性转义
// 富文本场景：使用 DOMPurify 等成熟库
```

**防御**：输出编码、`httpOnly` Cookie、CSP 头、`DOMPurify` 过滤。

### 7.2 CSRF（跨站请求伪造）

```
用户在 A 网站已登录 → 被诱导访问恶意网站 B →
网站 B 向 A 发出转账请求 → 浏览器自动携带 A 的 Cookie → 请求得逞
```

**防御**：
- **SameSite Cookie**（`Lax` / `Strict`）：最简便有效
- CSRF Token：表单隐藏字段 + Cookie 双写验证
- 验证 `Origin` / `Referer` 头
- 关键操作要求二次验证（验证码/密码）

### 7.3 Content Security Policy (CSP)

白名单机制，限制页面可加载/执行的资源来源。

```http
Content-Security-Policy:
    default-src 'self';
    script-src 'self' 'nonce-random123';
    style-src 'self' 'unsafe-inline';
    img-src * data:;
    font-src 'self' https://fonts.gstatic.com;
    connect-src 'self' https://api.example.com;
    frame-ancestors 'none';
    form-action 'self';
```

| 指令 | 作用 |
|---|---|
| `default-src` | 默认回退规则 |
| `script-src` | 可执行的 JS 来源 |
| `style-src` | 样式表来源 |
| `img-src` | 图片来源 |
| `connect-src` | fetch/WebSocket/EventSource 目标 |
| `frame-ancestors` | 哪些页面可嵌入本页（替代 X-Frame-Options） |

## 八、进程架构

现代浏览器（Chrome）为每个 tab 分配独立渲染进程，核心进程：

```
┌────────────────┐
│  Browser       │  主控：地址栏、书签、网络、文件访问（1 个）
│  Process       │
└──────┬─────────┘
       │
       ├── GPU Process        1 个，负责合成与 GPU 任务
       │
       ├── Network Process    1 个，处理所有网络请求
       │
       ├── Renderer Process   N 个（每个站点实例一个），沙箱隔离
       │    ├── 主线程：JS 执行、样式、布局、部分绘制
       │    ├── 合成线程：图层合成
       │    └── 光栅化线程池：将绘制指令转为位图
       │
       └── Plugin Process     N 个，Flash 等插件
```

关键含义：
- **一个 tab 崩溃不影响其他 tab**（进程隔离）
- **JS 执行占用主线程**，长任务会阻塞渲染和交互
- **合成在独立线程**，transform/opacity 动画不卡 JS
- 通过 `requestIdleCallback` 在空闲时段执行低优先任务

```javascript
// 非关键任务放到空闲回调
requestIdleCallback((deadline) => {
    while (deadline.timeRemaining() > 0 && tasks.length > 0) {
        const task = tasks.shift();
        task();
    }
}, { timeout: 2000 });  // 最迟 2 秒强制执行
```

## 九、总结速查

```
浏览器核心体系
├── 渲染流水线
│   ├── DOM + CSSOM → Render Tree → Layout → Paint → Composite
│   ├── CSS 阻塞渲染，同步 JS 阻塞解析
│   └── 动画：transform/opacity（仅合成）> color/background（重绘）> width/height（重排）
├── 缓存机制
│   ├── 强缓存：Cache-Control / Expires → 200 from cache
│   └── 协商缓存：ETag / Last-Modified → 304
├── 跨域处理
│   ├── CORS：服务端 Access-Control-* 响应头
│   ├── 代理：devServer.proxy / Nginx 反向代理
│   └── postMessage：Iframe 安全通信
├── 本地存储
│   ├── Cookie：4KB，随请求发送，httpOnly/SameSite 安全
│   ├── Web Storage：5MB，同步 API，纯 KV
│   └── IndexedDB：异步，大容量，结构化数据
├── 网络通信
│   ├── fetch + AbortController（现代方案）
│   ├── EventSource（单向推送）
│   └── WebSocket（双向实时）
└── 安全
    ├── XSS：不信任用户输入，CSP + 输出编码
    └── CSRF：SameSite Cookie + Token 验证
```
