# Redis 生产级实践模式

## 概述
本文讲解分布式锁、缓存策略、限流器、排行榜、幂等性等生产环境中最常用的设计模式。
每个模式的讲解包括：原理、实现方案对比、适用场景和常见坑。

前置知识：Redis 基本数据结构、Spring Boot 集成。

## 分布式锁

### 锁命令的演进

**第一代：SETNX（Redis 2.6.12 前）**

```
SETNX lock_key value    # 尝试获取锁
EXPIRE lock_key 30       # 设置过期时间
```

问题：SETNX 和 EXPIRE 是两步，不是原子的。如果 SETNX 成功后客户端崩溃，
锁永不过期（死锁）。

**第二代：SET NX EX（Redis 2.6.12+）**

```
SET lock_key value NX EX 30
```

NX + EX 在一条命令中完成，解决了死锁问题。但仍有问题：
过期时间设为多少？设短了业务没执行完锁就失效了，设长了其他请求等太久。

**第三代：锁续期（Redisson Watch Dog）**

Redisson 的看门狗机制：在锁即将过期前自动续期：

- 默认 leaseTime = 30 秒（不指定时）
- 每 leaseTime/3 = 10 秒检查一次
- 如果业务还在执行且锁还没释放 → 续期到 30 秒
- 业务执行完 → cancel 续期任务 + 释放锁

### 锁释放

释放锁不能简单 DEL——如果自己持有的锁已过期、锁被其他线程获取了，
DEL 就会误删别人的锁：

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

用 Lua 保证"判断锁属于自己"和"释放锁"是原子的。

### RedLock — 多节点锁

在 Redis 集群或 Sentinel 架构中，单个 master 挂掉会导致锁信息丢失。
RedLock 算法在 N 个独立节点上获取锁（通常 N=5）：

1. 获取当前时间
2. 依次向所有 N 个节点尝试获取锁
3. 锁的总获取时间 = 第二次获取时间 - 第一次获取时间
4. 如果获取了 N/2+1（多数）个节点的锁，且总时间 < 锁的 TTL，则获取成功
5. 锁的剩余有效时间 = TTL - 总获取时间

注意：RedLock 算法存在争议。Martin Kleppmann（Kafka 作者）指出，
在 GC stop、时钟跳跃等场景下 RedLock 的安全性保证不如 ZK。

### Spring 中使用 Redisson

Redisson 提供了 Java 中最完整的分布式锁实现：

```java
RLock lock = redissonClient.getLock("lock:order:" + orderId);

// 自动续期
lock.lock();

try {
    // 业务逻辑
} finally {
    lock.unlock();
}
```

## 缓存策略

### 四大缓存模式

```
Cache-Aside（旁路缓存 → 最常用）

    应用
    ├── 查缓存 → 命中返回
    ├── 未命中 → 查 DB → 写缓存 → 返回
    └── 写操作 → 写 DB → 删缓存

    缓存由应用代码控制，Redis 不感知 DB

Read-Through（读穿透）

    应用 → 缓存层 → 缓存未命中 → 缓存层自己查 DB → 返回
    应用不需要知道 DB 的存在
    实现：RedisGears / 自定义缓存中间件

Write-Through（写穿透）

    应用 → 缓存层 → 先写缓存 → 缓存同步写 DB → 返回
    保证缓存和 DB 一致但性能差
    每次写入都要等 DB 响应

Write-Behind（写回）

    应用 → 缓存层 → 写缓存就返回 → 异步批量写 DB
    性能和吞吐最高，但 DB 故障可能丢数据
    适合非关键数据（如浏览量）
```

实际生产中 **99% 的场景用的是 Cache-Aside**。

## 缓存三大问题

### 穿透

> 访问不存在的数据 → 缓存不命中 → 直接打到 DB

**方案一：布隆过滤器**

原理：用多个 hash 函数将元素的 hash 值映射到 bit array 的 k 个位置。
查询时如果 k 个位置都是 1，则认为元素存在。

特性：
- 可能存在（所有位都是 1）→ 允许通过 → 查 DB
- 一定不存在（有任意位是 0）→ 直接返回不存在
- 误报率 = (1 - e^(-kn/m))^k，m=位数组大小，n=预期元素数，k=hash 函数数
- 只能加不能删（删除需用 Counting Bloom Filter）

**方案二：缓存空值**

- 对查询到的空结果也缓存，value 是 null/空标记，TTL 较短（1-5 分钟）
- 简单但无法防御大量不同 key 的攻击（攻击者每次换一个 key）

### 击穿

> 热点数据过期瞬间 → 大量并发请求同时打到 DB

**方案一：互斥锁**

一个线程拿到锁去查 DB、写缓存，其他线程等待：

```
加锁(lock:hot:key)
  查缓存 → 未命中 → 查 DB → 写缓存 → 释放锁
等待获取锁
  读缓存 → 命中 → 返回
```

优点：强一致性。缺点：请求串行化，等待线程有延迟。

**方案二：逻辑过期**

缓存值中存储逻辑过期时间字段。物理 TTL 设很长（不触发自动过期）。
读取时发现逻辑过期：
- 返回旧值给用户（不阻塞）
- 异步起一个线程去更新缓存

优点：永不阻塞。缺点：返回可能过期的数据。

### 雪崩

> 大量缓存同时过期 → 瞬间流量全部打到 DB → DB 被打挂

**方案一：TTL 随机化**

给每个缓存 key 的 TTL 加上随机偏移（如 1 小时 ± 10 分钟），避免同时过期。

**方案二：多级缓存**

本地 Caffeine + Redis + DB 三级：
- 本地缓存 1-10 秒 → Redis 分钟级 → DB
- 一层挂了还有下一层兜底

**方案三：熔断降级**

- Hystrix/Sentinel 监控 DB 调用
- DB 超时/错误率超过阈值 → 直接返回默认值或拒绝服务
- 保护 DB 不被拖死

## 缓存更新一致性

这是一个没有完美答案的问题——在分布式系统中，缓存和 DB 的强一致性不可能做到 CAP 定理下的 CP（一致性+分区容错）。

### 四种更新策略对比

| 策略 | 流程 | 一致性问题 |
|------|------|-----------|
| Cache-Aside | 写 DB → 删缓存 | 读请求并发可能读到旧值并回写缓存 |
| Read-Through | 读缓存 → 缓存查 DB → 返回 | 同上 |
| Write-Through | 写缓存 → 缓存写 DB → 返回 | 写 DB 前线程崩溃，缓存和 DB 不一致 |
| Write-Behind | 写缓存就返回 → 异步写 DB | DB 更新失败 → 丢数据 |

### 更新顺序的经典争议

**「先删缓存再写 DB」**：
- 问题：删缓存 → 写 DB（未完成）→ 读请求（DB 返回旧值→回写缓存脏数据）
- 解决：延迟双删（写 DB 完成后，sleep 几百 ms，再删一次缓存）

**「先写 DB 再删缓存」**：
- 问题：缓存未命中 → 读 DB（返回旧值）→ 写 DB（新值）→ 删缓存 → 读请求的旧值回写缓存
- 概率极低但理论上存在
- 业界多数采用此方案（概率低 + 代价小）

### 最终一致方案（推荐）

```
写 DB → 发 binlog → Canal/Debezium 监听 → 异步 MQ → 消费者更新缓存
```

代价：系统复杂、有延迟（通常秒级）。但能覆盖绝大多数场景。

## 限流器

### 四种算法

**固定窗口**：
- 计数器 INCR + TTL=窗口时长
- 简单但有关键问题——窗口边界时请求翻倍

**滑动窗口（ZSet 实现）**：
- ZSet member=唯一请求标识，score=时间戳
- 每次请求：ZREMRANGEBYSCORE 清除窗口外 → ZADD 当前请求 → ZCARD 判断
- 精确但有内存开销

**令牌桶**：
- 定时以固定速率添加令牌（用 Lua 脚本或独立的定时任务）
- 请求时取走一个令牌
- 允许突发（桶满了就存着令牌）

**漏桶**：
- 请求进队列，以固定速率取出处理
- 消费速率固定，无法处理突发

选择：Java 侧一般用 Guava RateLimiter（令牌桶），Redis 侧用 Lua 实现滑动窗口。

## 排行榜

### ZSet 排行榜

```
ZADD leaderboard score member       # 更新分数
ZREVRANGE leaderboard 0 9           # Top 10
ZREVRANK leaderboard member         # 查用户排名（0 开始）
ZREVRANGE leaderboard 0 9 WITHSCORES # 带分数
```

### 复合排名

score 复合设计：`分数 * 10^10 + (maxTime - 时间戳)`

这样分数相同的，时间早的排在前面。

### 亿级排行榜

- 分桶：uid % 100 分到 100 个 ZSet 中
- 查询某一用户排名：用户所在桶内排名 + 所有桶内分数比用户高的总人数
- 查询 Top N：每个桶取 Top N，合并排序取 Top N

## SCAN 替代 KEYS

### KEYS 的问题

KEYS 是 O(N) 遍历所有 key（N=整个 Redis 的 key 总数），且执行期间阻塞单线程。
生产环境 100 万 key 的 KEYS 可能耗时 200ms+，期间所有客户端请求都等待。

### SCAN 原理

```
SCAN 0 COUNT 100
→ 返回下个游标 + 一批 key

SCAN <返回的游标> COUNT 100
→ 继续...
→ 游标回到 0 表示结束
```

关键点：
- COUNT 不是返回数量，是提示单个分桶的大小
- 可能返回空结果，但游标不为 0（桶很大但全是空槽）
- 可能返回重复 key（rehash 期间一个 key 可能在两个 hash 表中都有）
- SCAN 是增量迭代，不阻塞

## 幂等性

### SetNX 幂等 Token

```
请求到达
  → 生成唯一 token（如订单号）
  → SetNX idempotent:token 1 EX 300  # 5 分钟窗口
  → 成功：执行业务 → 返回
  → 失败：已有相同请求在处理 → 返回重复提交
```

TTL 选择：比业务最长执行时间长一点，防止锁过期后重复提交。

---

## 总结

1. **分布式锁用 Redisson**，别自己手写 SET NX EX 轮子
2. **缓存三大问题的本质**：穿透 = 不存在的数据打到 DB，击穿 = 单个热 key 失效，雪崩 = 大量 key 同时失效
3. **缓存与 DB 一致性没有完美方案**，大多数场景选「先写 DB 再删缓存」
4. **生产环境禁用 KEYS**，用 SCAN 替代
5. **幂等性是分布式系统中必须考虑的问题**，SetNX + TTL 是最简单的方案
