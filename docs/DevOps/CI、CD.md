# 什么是 CI/CD？

CI/CD 是现代软件开发中的一套重要工程实践，主要包括：

- CI（Continuous Integration，持续集成）
- CD（Continuous Delivery，持续交付）
- CD（Continuous Deployment，持续部署）

其核心目标是通过自动化手段提升软件开发效率，降低人为操作带来的风险，加快代码从开发到生产环境的交付速度，并持续保证软件质量与稳定性。

在传统开发模式下，开发、测试与运维往往相互割裂。开发人员编写代码后，通常需要等待统一测试、统一部署，整个流程严重依赖人工操作。这种方式在多人协作时容易出现代码冲突、环境不一致、上线失败等问题，而且问题往往会在项目后期才暴露，修复成本较高。CI/CD 的出现，本质上就是为了让软件交付流程更加自动化、标准化和持续化。

---

# CI 持续集成（Continuous Integration）

持续集成强调开发人员频繁地将代码提交并合并到公共代码仓库中，而不是长时间在本地或个人分支上独立开发。每次代码提交后，系统会自动执行编译、单元测试、集成测试、代码检查等操作，从而尽早发现问题。

在多人协作开发中，如果多个开发人员长期不合并代码，很容易在后期产生大量冲突，甚至导致系统功能异常。而持续集成通过“小步快跑”的方式，让问题尽可能在早期暴露。例如某次提交导致接口异常、依赖冲突或单元测试失败，CI 系统会立即反馈给开发人员，此时修改范围小、定位清晰，修复成本也最低。

因此，CI 的核心思想并不仅仅是“自动化测试”，更重要的是：

> 尽早发现问题，尽早修复问题。

通常在 CI 流程中，会自动完成以下内容：

- 拉取最新代码
- 编译构建项目
- 执行自动化测试
- 进行静态代码分析
- 检查安全漏洞
- 生成构建产物

常见的 CI 工具包括 Jenkins、GitHub Actions、GitLab CI/CD 等。

---

# CD 持续交付（Continuous Delivery）

持续交付是在持续集成基础上的进一步扩展。CI 主要解决“代码能否稳定集成”的问题，而持续交付更关注：

> 软件是否已经具备随时发布到生产环境的能力。

在持续交付流程中，代码在通过自动化测试后，会继续完成打包、构建镜像、上传制品仓库、部署测试环境等操作。也就是说，系统已经自动完成了上线前的大部分工作，运维人员或负责人只需要进行一次人工确认，就可以将系统部署到生产环境。

持续交付的重点是让“发布”变得简单、稳定且低风险。开发团队无需再依赖复杂的人工部署流程，也不需要在上线前临时处理各种环境问题。整个系统始终处于“可以随时上线”的状态。

例如在 Spring Boot 微服务项目中，持续交付通常会自动完成：

- Maven/Gradle 构建
- Docker 镜像生成
- 镜像推送到 Harbor 或 Docker Hub
- 自动部署测试环境
- 自动化验收测试

最终由人工决定是否正式上线。

---

# CD 持续部署（Continuous Deployment）

持续部署相比持续交付自动化程度更高，它强调：

> 只要代码通过所有测试，就自动部署到生产环境。

也就是说，在持续部署模式下，不再需要人工审核或手动点击发布按钮。开发人员提交代码后，系统会自动完成测试、构建、部署，并直接上线供用户使用。

持续部署能够显著提升软件迭代速度，特别适合互联网高频更新场景。开发团队可以快速获得真实用户反馈，并及时修复问题或调整功能。

不过，持续部署对自动化测试体系要求极高。如果测试覆盖率不足，错误代码可能会直接进入生产环境。因此，一些对稳定性要求极高的系统，例如金融、医疗、支付等领域，通常更倾向于采用“持续交付”而不是完全自动化的持续部署。

---

# CI、持续交付与持续部署的关系

CI/CD 并不是几个独立概念的简单组合，而是一整套完整的软件交付流水线。

其中，CI 负责解决代码集成与质量验证问题；持续交付负责让系统始终具备可发布能力；持续部署则进一步实现自动上线。

三者通常是逐步演进的关系：

```
持续集成（CI）        
	↓
持续交付（Continuous Delivery）        
	↓
持续部署（Continuous Deployment）
```

一般来说，一个团队会先建立稳定的 CI 流程，在代码能够稳定集成后，再逐渐推进自动化部署与发布。

---

# CI/CD 与 DevOps 的关系

CI/CD 是 DevOps 中非常核心的一部分，但 DevOps 并不等同于 CI/CD。

CI/CD 更偏向于工程自动化流程，而 DevOps 更强调开发与运维之间的协作文化，以及软件生命周期的整体优化。除了 CI/CD 外，DevOps 还包括：

- 自动化运维
- 容器化
- Kubernetes 编排
- 基础设施即代码（IaC）
- 日志与监控
- 灰度发布
- 蓝绿部署
- 自动化扩缩容

因此，可以将 CI/CD 理解为 DevOps 落地实践中最核心的一条自动化流水线。

---

# 总结

持续集成（CI）主要解决代码如何稳定、高频地合并问题；持续交付关注如何让系统始终保持“可发布”状态；持续部署则进一步实现自动上线。

简单来说：

```
CI：自动测试与集成代码
持续交付：自动准备上线
持续部署：自动完成上线
```

它们共同构成了现代软件工程中最重要的自动化交付体系之一。
