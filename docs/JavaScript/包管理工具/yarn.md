# 为什么会出现 Yarn

早期，前端项目大多使用 Node.js 自带的 npm 作为包管理工具。

但在 2016 年前后，随着前端工程化快速发展，项目依赖数量越来越多，npm 在大型项目中的一些问题开始逐渐暴露，例如：

- 安装速度较慢
    
- 依赖下载不稳定
    
- `node_modules` 层级复杂
    
- 多人协作时依赖版本容易不一致
    
- 缺少可靠的锁文件机制
    
- 某些情况下会出现“我电脑能运行，你电脑却无法运行”的问题
    

尤其是在大型前端项目中，大量依赖嵌套后，依赖树会变得越来越复杂，npm 的安装结果在不同机器、不同时间下可能并不一致。

于是，Facebook 联合 Google、Exponent（后来的 Expo）等团队，于 2016 年推出了 Yarn。

Yarn 的核心目标主要包括：

- 提升依赖安装速度
    
- 保证依赖版本一致性
    
- 优化网络下载稳定性
    
- 改善依赖解析体验
    

---

# Yarn 的重要特性

## yarn.lock（锁文件）

例如：

```json
{
  "dependencies": {
    "react": "^18.0.0"
  }
}
```

这里的 `^18.0.0` 实际允许安装：

- 18.0.1
    
- 18.1.0
    
- 18.2.0
    

等兼容版本。

这意味着：

不同时间执行安装命令，最终得到的依赖版本可能不同。

为了解决这一问题，Yarn 引入了 `yarn.lock` 锁文件，用于记录：

- 每个依赖的真实版本
    
- 下载地址
    
- 依赖关系
    
- 校验信息
    

从而保证团队成员安装出的依赖结果一致。

后续 npm 也借鉴了这一思路，并推出了 `package-lock.json`。

---

## 并行下载

早期 npm 在安装依赖时，效率相对较低。

Yarn 对安装流程进行了大量优化，例如：

- 并发请求
    
- 并行下载
    
- 本地缓存复用
    
- 网络请求优化
    

因此在当时，`yarn install` 通常比 `npm install` 更快。

---

## 离线缓存

Yarn 会缓存已经下载过的依赖包。

后续再次安装时，可以直接复用缓存内容，部分情况下甚至能够实现离线安装。

这不仅提高了安装速度，也减少了网络不稳定带来的影响。

---

## 更可靠的依赖解析

早期 npm 在某些情况下可能会出现：

- 依赖树不稳定
    
- 依赖提升异常
    
- 幽灵依赖
    

等问题。

其中，“幽灵依赖”指的是：

> 项目没有显式安装某个依赖，但由于其他包间接安装，代码仍然能够运行。

这种情况容易导致：

- 换一台机器后项目无法运行
    
- 不同开发环境结果不一致
    

Yarn 在依赖解析与安装机制上进行了大量优化，使依赖结构更加稳定和可预测。

---

# Yarn 的版本演进

## Yarn 1（Classic）

Yarn 1 是最经典、兼容性最好的版本。

特点：

- 使用 `node_modules`
    
- 与 npm 生态兼容性高
    
- 上手简单
    
- 社区使用广泛
    

---

## Yarn 2+（Berry）

Yarn 2 之后，Yarn 进行了较大重构，也被称为 Yarn Berry。

其主要特性包括：

- Plug'n'Play（PnP）
    
- 更严格的依赖管理
    
- 更快的安装速度
    
- 更现代化的工作区支持
    

其中 PnP 默认不再依赖 `node_modules`，而是通过特殊的依赖映射机制直接解析包。

虽然性能更好，但部分旧项目或工具链可能存在兼容性问题。

---

# npm 后续的发展

随着 npm 5+、npm 7+ 的持续发展，npm 也逐渐引入了：

- 锁文件机制
    
- 本地缓存
    
- 并行下载
    
- Workspaces
    
- 更稳定的依赖解析
    

因此如今 npm 与 Yarn 的差距已经明显缩小。

目前前端生态中常见的包管理工具主要包括：

- npm
    
- Yarn
    
- pnpm
    

其中 pnpm 因磁盘占用更低、安装速度更快，近年来也越来越流行。

# 常用命令

## 初始化项目

```bash
yarn init
```

---

## 安装依赖

```bash
yarn
```

或：

```bash
yarn install
```

默认情况下，直接执行 `yarn` 等价于 `yarn install`。

---

## 添加依赖

安装最新版本：

```bash
yarn add react
```

安装指定版本：

```bash
yarn add react@18.2.0
```

安装指定主版本范围：

```bash
yarn add react@^18
```

安装指定次版本范围：

```bash
yarn add react@~18.2.0
```

安装某个标签版本：

```bash
yarn add react@latest
```

```bash
yarn add react@beta
```

安装开发依赖：

```bash
yarn add vite -D
```

或：

```bash
yarn add vite --dev
```

---

### 常见版本符号说明

|符号|示例|含义|
|---|---|---|
|`^`|`^18.2.0`|允许升级次版本和补丁版本|
|`~`|`~18.2.0`|仅允许升级补丁版本|
|`*`|`*`|匹配任意版本|
|`latest`|`@latest`|安装最新稳定版本|

例如：

```json
{
  "dependencies": {
    "react": "^18.2.0"
  }
}
```

可能实际安装：

- 18.2.1
    
- 18.3.0
    

但不会升级到：

- 19.0.0
    

---

## 删除依赖

```bash
yarn remove react
```

`remove` 删除的是整个依赖包，因此不需要指定版本号。

---

## 升级依赖

升级全部依赖：

```bash
yarn upgrade
```

升级指定依赖：

```bash
yarn upgrade react
```

---

## 执行脚本

```bash
yarn dev
```

```bash
yarn build
```

```bash
yarn start
```

等价于：

```bash
npm run dev
npm run build
npm run start
```

Yarn 可以省略 `run` 关键字，因此命令更加简洁。

---

## 查看版本

```bash
yarn --version
```

---

# 总结

Yarn 的出现，本质上是前端工程化发展的产物。

它解决了早期 npm 在：

- 安装速度
    
- 依赖一致性
    
- 网络稳定性
    
- 依赖解析
    

等方面的问题，并推动了整个 JavaScript 包管理生态的发展。

后续 npm 与 pnpm 也不断吸收 Yarn 的优秀设计理念，使现代前端包管理工具越来越成熟。
