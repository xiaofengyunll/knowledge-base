# 前端性能优化技术指南

从浏览器渲染、资源加载、网络请求到打包构建，系统性地梳理前端性能优化的核心手段与决策依据。

---

## 一、页面渲染优化

页面渲染性能的核心指标围绕两个问题展开：**首屏内容多久能出来**（FP/FCP/LCP），以及**页面交互跟不跟手**（INP）。优化的主战场在浏览器的主线程——脚本执行、样式计算、布局、绘制全部挤在这条线上。

### 1.1 关键渲染路径（Critical Rendering Path）

浏览器从收到 HTML 到绘制出像素，走的是固定的流水线：

```
HTML → DOM Tree
CSS  → CSSOM Tree
       ↓
   Render Tree → Layout → Paint → Composite
```

DOM 和 CSSOM 的构建是并行的，但两者都就绪之前渲染树无法生成。这意味着：

- HTML 文档流中的 CSS（`<link>` 或 `<style>`）会阻塞渲染，但不会阻塞 DOM 构建。
- 位于 `<head>` 中同步加载的 JS 会阻塞 DOM 构建和后续的 CSSOM 构建——脚本可能操作 DOM，浏览器必须暂停解析直到脚本执行完毕。

**实践方向：**

- 将关键 CSS（首屏可见内容的最小样式集）内联到 `<head>` 的 `<style>` 标签中，其余 CSS 以非阻塞方式加载。
- 将非关键 JS 标记为 `async`（不阻塞解析，下载完立即执行）或 `defer`（不阻塞解析，DOM 就绪后按顺序执行）。
- 避免在 `<head>` 中出现同步 `<script>` 标签，除非确实需要在 DOM 构建前介入逻辑。

```html
<!-- 关键 CSS：内联，零额外请求 -->
<style>
  /* 首屏布局与字体 */
</style>

<!-- 非关键 CSS：异步加载 -->
<link rel="preload" href="/styles/full.css" as="style" onload="this.rel='stylesheet'">

<!-- JS：defer 保证按顺序在 DOM 就绪后执行 -->
<script defer src="/app.js"></script>
```

### 1.2 避免布局抖动（Layout Thrashing）

在一帧内交替执行"读取布局信息"和"修改样式"的操作，会导致强制同步布局（forced synchronous layout），极端情况下每一对操作都触发一次布局重算，帧时间直接爆炸。

```javascript
// 典型的布局抖动：读-写-读-写交叉
for (const el of elements) {
  const height = el.offsetHeight; // 读 → 触发布局
  el.style.height = height + 1 + 'px'; // 写 → 脏标记
  // 下一轮循环的 offsetHeight 在读之前又触发一次强制布局
}
```

**修正方式：** 将"读"操作和"写"操作分别集中到两个阶段。

```javascript
// 阶段一：批量读
const heights = elements.map(el => el.offsetHeight);
// 阶段二：批量写
elements.forEach((el, i) => {
  el.style.height = heights[i] + 1 + 'px';
});
```

现代框架（React 虚拟 DOM、Vue 的异步 DOM 更新队列）在大多数场景下自动规避了这个问题，但直接操作 DOM 的代码（动画、第三方脚本、自定义指令）仍然是高风险区域。排查时可在 Chrome DevTools → Performance 面板中寻找标记为 "Forced reflow" 的记录。

### 1.3 GPU 加速与合成层

浏览器的渲染可以简化为三层：**绘制**（Paint）→ **合成**（Composite）→ **显示**。如果只改变 `transform` 或 `opacity`，浏览器可以跳过重排和重绘，直接走合成，这是实现 60fps 动画的基础。

但滥用 `will-change` 或 `transform: translateZ(0)` 强行提升合成层会导致显存膨胀——每个合成层都分配独立的 GPU 纹理。移动端尤其敏感，过多合成层可能直接引发崩溃。

```css
/* 正确用法：仅在动画前提升，动画结束后移除 */
.animate-ready {
  will-change: transform;
}
.animate-done {
  will-change: auto;
}
```

判断是否过度使用合成层的直观指标：DevTools → Layers 面板中看到大量非必要的独立图层，或者在 Rendering 面板中开启 "Layer borders" 看到满屏橙色边框。

### 1.4 长任务拆分

单次超过 50ms 的长任务（Long Task）直接阻塞主线程，用户输入无法得到响应。React 18 的 Concurrent Mode 和 Scheduler 在框架层面做了时间切片，但没有框架兜底的纯计算场景需要手动拆分：

```javascript
// 将大计算量任务拆分成多个 micro-task
function processInChunks(items, chunkSize, callback) {
  let i = 0;
  function chunk() {
    const end = Math.min(i + chunkSize, items.length);
    for (; i < end; i++) {
      // 处理 items[i]
    }
    if (i < items.length) {
      requestIdleCallback(chunk); // 或使用 scheduler.postTask
    } else {
      callback();
    }
  }
  chunk();
}
```

`requestIdleCallback` 在浏览器空闲时执行，适合非紧急的后台任务。需要更高优先级的任务可以使用 `scheduler.postTask`（Chrome 支持，有 polyfill）。

### 1.5 虚拟列表

渲染数千条 DOM 节点的开销不在 JS 循环，在浏览器——每个 DOM 节点都有内存开销、样式计算开销和合成层开销。虚拟列表通过只渲染可视区域及其缓冲区的少量节点来规避这个问题。

具体选型取决于场景：`react-window`（轻量，适合固定高度）、`react-virtuoso`（动态高度开箱即用）、手写方案（需要精确的滚动容器测量）。核心不是库本身，而是理解滚动偏移、容器高度、行高的三角关系。

## 二、资源加载优化

资源加载策略的核心矛盾是：浏览器需要尽早发现关键资源，但又不能被非关键资源挤占带宽。

### 2.1 资源优先级与 Preload/Prefetch

浏览器对资源的默认优先级按类型排序：CSS（Highest）> 同步 JS（High）> 字体（High）> 图片（Low）。但这个排序不一定符合业务的真实需求——首屏的大图可能被排在低优先级，导致 LCP 滞后。

`<link rel="preload">` 明确告诉浏览器"这个资源很重要，现在就下载"，它会打断默认的优先级排序：

```html
<!-- LCP 关键图片：提升到最高优先级 -->
<link rel="preload" as="image" href="/hero.webp">

<!-- 关键字体：浏览器在发现 font-face 规则后才开始下载，太晚了 -->
<link rel="preload" as="font" href="/fonts/Inter.woff2" crossorigin>
```

`preload` 与 `prefetch` 的区别：`preload` 用于当前页面的关键资源；`prefetch` 用于**下一个页面可能用到**的资源，下载优先级最低，浏览器空闲时才执行。

实践中容易犯的错误：preload 用太多导致优先级系统被稀释——所有资源都是高优先级等于没有优先级。preload 应聚焦在 2-3 个 LCP 关键资源上。

### 2.2 图片优化

图片在很多业务中占到页面传输体积的 50% 以上。优化的三个维度：

**格式选择。** WebP 相比 JPEG 在同等质量下减少 25%-35% 体积，支持无损和 alpha 通道。AVIF 更进一步，但编码速度较慢，适合对构建时间不敏感的静态资源。HEIF/HEIC 在 iOS 生态中广泛使用，浏览器支持仍有限。实际策略：给出 WebP/Avif 格式，配合 `<picture>` 降级 JPEG/PNG。

```html
<picture>
  <source srcset="/hero.avif" type="image/avif">
  <source srcset="/hero.webp" type="image/webp">
  <img src="/hero.jpg" alt="..." loading="lazy" decoding="async">
</picture>
```

**尺寸控制。** `srcset` + `sizes` 使浏览器根据设备宽度和 DPR 选择最合适的尺寸，避免在移动端加载 2x 大图。图片 CDN（imgix、Cloudinary、阿里云/腾讯云图片处理）可以在 URL 参数中指定宽高和裁切策略，服务端实时处理，前端免去预生成多尺寸的构建复杂度。

**加载策略。** `<img loading="lazy">` 原生懒加载已经覆盖多数场景。需要更精细控制（如低质量占位图渐进加载）时使用 `IntersectionObserver`：

```javascript
const observer = new IntersectionObserver((entries) => {
  entries.forEach(entry => {
    if (entry.isIntersecting) {
      const img = entry.target;
      img.src = img.dataset.src;
      observer.unobserve(img);
    }
  });
});
images.forEach(img => observer.observe(img));
```

### 2.3 字体加载优化

Web 字体是 FOUT（无样式文本闪烁）和 FOIT（文本不可见）的根源。默认行为下，浏览器在字体下载完成前不渲染文字，白屏时间直接拉长。

关键策略：`font-display: swap` 让浏览器先用回退字体渲染文本，字体下载完成后再切换——用户能立刻看到内容，虽然字体切换时会有短暂的视觉跳动。

```css
@font-face {
  font-family: 'Inter';
  src: url('/fonts/Inter.woff2') format('woff2');
  font-display: swap;
}
```

对于需要视觉一致性胜过速度的场景（品牌页面），使用 `font-display: optional`——字体在 100ms 内下载不到就永久使用回退字体，避免页面加载过程中发生的布局偏移。

子集化是另一个有效的体积缩减手段：一个完整的中文字体文件可达 3-10MB，但页面实际使用的中文字符可能只有数百个。在构建时使用 `fontmin` 或在线服务裁剪字体到页面引用的字符集，体积通常能降到 50-200KB。

### 2.4 第三方脚本的隔离

第三方脚本（监控、广告、客服聊天）是前端性能中最难控制的变量——它们不在你的构建流程里，随时可能阻塞主线程或被阻断影响整个页面的渲染。

防御措施：用 `<script type="module">` 天然获得延迟执行特性；对于不能模块化的第三方脚本，使用 Facade 模式——先用一个静态的、点击后才激活真实脚本的按钮替代：

```html
<!-- Facade：用一个假的聊天按钮替代即时加载的第三方脚本 -->
<button id="chat-facade">在线客服</button>
<script>
  document.getElementById('chat-facade').addEventListener('click', () => {
    // 点击后才加载第三方聊天脚本
    const s = document.createElement('script');
    s.src = 'https://third-party.com/chat.js';
    document.head.appendChild(s);
  });
</script>
```

## 三、网络请求优化

网络这部分，对页面加载性能影响最大的通常是"请求链路的长度"——RTT 的累积才是真正拖慢首屏的因素，而不是传输带宽本身。

### 3.1 HTTP 协议升级

**HTTP/2 的多路复用**消除了 HTTP/1.1 的队头阻塞（同一连接上的请求必须串行完成），允许在一个 TCP 连接上并发传输多个请求/响应，请求之间相互不影响。以往依赖的"域名分片"（把资源分布到多个域以增加并行连接数）在 HTTP/2 下反而有害——多域名意味着额外的 DNS 查询、TCP 握手和 TLS 协商。

**HTTP/3（QUIC）**更进一步，把传输层从 TCP 换成了 UDP。带来的直接好处是：连接建立只需 0-RTT（之前访问过该站点时）；丢包只影响单个流而不阻塞其他流，解决了 HTTP/2 在 TCP 层面的队头阻塞问题。从 HTTP/2 升级到 HTTP/3，在高丢包网络（移动网络）上的页面加载时间可减少 10%-20%。

升级路径：服务端（Nginx 1.25+、Caddy、Cloudflare）开启 HTTP/3 支持，客户端零改造成本——浏览器自动在首次访问时通过 Alt-Svc 头发现 HTTP/3 可用性。

### 3.2 减少请求量

即使有 HTTP/2 的多路复用，请求数量仍然是影响因素——每个请求都有头部开销和优先级调度开销。

**资源合并的权衡。** HTTP/1.1 时代流行的 CSS sprites、JS/CSS 全量合并，在 HTTP/2 时代收益不再显著，反而因为缓存粒度变粗（修改一行 CSS 就得整个 bundle 重新下载）而导致缓存命中率下降。一个更合理的折中：将变动的代码与稳定的第三方库代码分开打包，利用 HTTP/2 并行加载各自的 chunk。

**接口聚合。** 在 SPA 的首屏加载中，往往出现 10+ 个接口并发请求，虽然请求本身并行，但各个接口都独立走完认证、查库、响应的完整链路，首包时间受最慢的接口限制。BFF（Backend For Frontend）层的价值就在这里——为每个页面定制一个聚合接口，将分散的请求合并为一次端到端调用，消除多个独立请求的延迟累积。

**GraphQL 是这个思路的自然延伸**，允许客户端在一次请求中精确声明需要的数据字段，缺点是服务端的解析和嵌套解析开销不可忽视，简单 CRUD 场景下可能得不偿失。

### 3.3 缓存策略

**浏览器缓存（强缓存/协商缓存）。** 强缓存（`Cache-Control: max-age`）在有效期内完全跳过请求，协商缓存（`ETag`/`Last-Modified`）每次仍然发出请求但可能返回 304。对于带有 hash 的文件名（`app.a3f2.js`），可以将 `max-age` 设为一年，因为内容变化会改变文件名从而打破缓存。对于 HTML 文档本身，建议用 `Cache-Control: no-cache` 确保每次都验证——HTML 是其他资源的入口，一旦被缓存过久而无法更新，引用的资源路径也会卡在老版本。

**Service Worker 缓存。** SW 提供了一个程序化的缓存层，可以在请求发出前拦截并返回缓存的响应，这是实现离线可用和 App Shell 架构的基础。但 SW 也是"最难删除的 bug 来源"之一——缓存策略错误会导致用户永远看到旧版本。Workbox 库封装了常见的缓存策略：

- **Cache First**：适合不变的静态资源（带 hash 的 JS/CSS）
- **Network First**：适合可能变化的 API 响应
- **Stale While Revalidate**：适合非关键资源——立即返回缓存，同时异步更新

```javascript
// Workbox 路由示例
registerRoute(
  ({ request }) => request.destination === 'style',
  new StaleWhileRevalidate({ cacheName: 'css-cache' })
);
```

**Memory Cache 与 Disk Cache。** 浏览器有两级缓存：内存缓存（更快、容量小、页面关闭即释放）、磁盘缓存（持久化、容量大）。用户在同一页面内回退导航时，浏览器优先从 Memory Cache 读取，这是 SPA 路由切换体验流畅的原因之一。但在实际优化中，开发者不需要、也无法精确控制这一层——应该把精力放在 HTTP 缓存头的配置上。

### 3.4 数据预取与"预测"

从用户在页面上 hover 一个链接，到真正点击，再到目标页面加载，中间有数百毫秒的窗口可以利用：

- **`<link rel="prefetch">`**：浏览器在空闲时下载下一个页面需要的 JS/CSS
- **`<link rel="prerender">`**（Chrome）：预渲染完整的下一页面，用户点击后瞬间切换

预取的边界是：你猜错了就会浪费带宽和计算资源。更可靠的做法是结合业务的导航路径——用户在当前页面停留 3 秒后预取最常跳转的目标页，或者 hover 一个链接 100ms 后预取该链接。

### 3.5 gzip/Brotli 压缩

Brotli 是 Google 开发的压缩算法，在文本资源（HTML/CSS/JS/JSON/SVG）上的压缩率比 gzip 高 20% 左右。几乎所有现代 CDN 都支持 Brotli，服务端开启后客户端通过 `Accept-Encoding: br` 自动协商。对于 JS bundle 这种体积大户，Brotli 的压缩优势尤其明显。

注意：Brotli 的最高压缩级别在服务端消耗大量 CPU，动态资源建议使用较低的压缩级别（1-4），静态资源可以在构建时预压缩到最高级别（11）。

## 四、打包构建优化

打包优化直接影响到用户下载的 JS/CSS 体积，也间接决定了浏览器解析和执行 JS 的时间。

### 4.1 Code Splitting 与 Tree Shaking

**Code Splitting** 的思路是把"首屏需要的代码"和"后续才需要的代码"分开。Webpack 的 `import()` 和 React 的 `React.lazy()` 在底层做的是同一件事——将动态导入的模块及其依赖打包为独立 chunk，浏览器在真正需要时才发起请求。

切分的粒度需要权衡：太粗（一个巨大的 vendor chunk + 一个 app chunk），首屏仍然加载了大量非必要代码；太细（每个路由一个 chunk），可能出现 50+ 个小文件，HTTP/2 下虽然可以并行下载，但每个文件仍有压缩效率降低和调度开销。实际项目中，首屏的路由入口单独成 chunk；第三方库按体积和改动频率分成 stable vendor 和偶尔更新的 ui lib；公共的工具函数和组件提取为 common chunk。

**Tree Shaking** 依赖 ES Module 的静态导入（`import { fn } from './utils'`），这让打包工具在编译期就能确定哪些导出被使用、哪些可以丢弃。但 Tree Shaking 在实际项目中经常失效，原因包括：

- 第三方库以 CommonJS 发布而非 ESM（打包工具无法静态分析 CJS）
- 副作用代码被标记为 `sideEffects: false` 之后仍引入了全局样式或 polyfill
- Babel 的 `modules: "auto"` 配置将 ESM 转成了 CJS，Tree Shaking 失去静态特征

排查 Tree Shaking 是否生效的直接办法：在 Webpack 配置中打开 `optimization.usedExports: true`，构建后检查输出的 JS 文件，搜索预期被 tree-shake 掉的函数名。

### 4.2 懒加载的三种模式

**路由级懒加载。** SPA 最基本的分割方式，每个路由对应一个 chunk：

```javascript
const Dashboard = React.lazy(() => import('./pages/Dashboard'));
const Settings = React.lazy(() => import('./pages/Settings'));
```

**组件级懒加载。** 适用于体积大但用户不一定触发的组件——重型的图表库、编辑器、视频播放器。原则是"不看到（或不即将看到）就不加载"：

```javascript
const HeavyEditor = React.lazy(() => import('./components/Editor'));

function Page() {
  const [showEditor, setShowEditor] = useState(false);
  return (
    <div>
      <button onClick={() => setShowEditor(true)}>编辑</button>
      {showEditor && (
        <Suspense fallback={<Skeleton />}>
          <HeavyEditor />
        </Suspense>
      )}
    </div>
  );
}
```

**库级懒加载。** 某些工具函数的体积很大但只在特定交互后才需要。例如 PDF 生成只在用户点击"导出"时触发：

```javascript
async function exportPDF() {
  const { jsPDF } = await import('jspdf');
  const doc = new jsPDF();
  // ...
}
```

### 4.3 构建产物的分析

`webpack-bundle-analyzer` / `rollup-plugin-visualizer` 是优化前的第一步——你需要看到当前产物的构成，才能定位"最大的一刀该切在哪里"。常见的意外发现：

- Moment.js 引入了全套 locale（+ 200KB）
- 同一个 lodash 函数在两个 chunk 中重复出现，因为 common chunk 提取策略不对
- 某图标库全量引入而非按需引入，图标体积占总 bundle 的 40%

`source-map-explorer` 提供了另一种视角：直接分析 source map，展示原始文件代码在最终产物中的体积占比，比单纯的模块体积报表更贴近实际情况。

### 4.4 构建速度

构建速度不直接影响用户，但影响开发和发布效率。关键手段：

- **持久化缓存。** Webpack 5 的 `cache: { type: 'filesystem' }` 将中间产物写入磁盘，增量构建从数十秒降到 1-3 秒。Vite 利用 esbuild 的预构建和浏览器原生 ESM 让开发服务器冷启动在毫秒级。
- **编译器选择。** esbuild（基于 Go）和 SWC（基于 Rust）在编译速度上是 Babel 的 10-100 倍。生产构建中，用 esbuild-loader 或 swc-loader 替换 babel-loader 可以获得显著提升，代价是某些 Babel 插件的生态尚未完全迁移——需要检查项目中是否依赖了尚未被 SWC 支持的 Babel 插件。
- **模块解析。** Webpack 的 `resolve.modules` 和 `resolve.extensions` 配置应尽量精确——`extensions: ['.js', '.ts', '.tsx']` 而非 `['.js', '.json', '.wasm', '.ts', '.tsx', '.jsx', '.vue', '.mjs']`——每个未命中的扩展名尝试都是一次不必要的文件系统查找。
- **CI 并行化。** 将构建任务拆分为 npm workspace 独立构建子包，或使用 Nx/Turborepo 的分布式任务执行能力，让多个构建步骤在同一台构建机或分布式集群中并行执行。
- **依赖管理。** `pnpm` 的引用式安装节省磁盘空间，但更关键的是它的 node_modules 结构防止了幽灵依赖——项目不会隐式依赖未在 package.json 中声明的包，降低了构建不确定性和"在我机器上能跑"的几率。

## 五、常见误区

**"优化就是压缩图片和合并 JS。"** 图片和 JS 体积只是带宽侧的优化，首屏时延的瓶颈更可能在关键渲染路径上的阻塞资源、服务端的响应延迟、或者 JavaScript 执行时间对交互的阻塞。优化应该从测量开始，而不是从手段开始。

**"开启 HTTP/2 之后之前的所有优化都没用了。"** HTTP/2 多路复用解决了并发限制，但并未消除资源加载的延迟——把一个 2MB 的 bundle 拆成 20 个 100KB 的 chunk，每个 chunk 仍然需要各自完成 DNS + TCP + TLS（或 QUIC 握手），只是它们可以同时进行。带宽和延迟仍然是物理约束。

**"加了 `React.lazy` 就算做了代码分割。"** 懒加载降低了首屏的 JS 下载量，但如果用户首屏触发的路由交互恰好是所有懒加载 chunk 中最重的那个，首屏体验并没有实质改善。代码分割策略必须跟业务的路由和用户行为分布对齐。

**"Core Web Vitals 全绿就代表性能没问题。"** CWV 测量的是用户体验的底线，不是上限。一个 LCP 2.4 秒（刚好在 2.5 秒"良好"线内）的页面，和一个 LCP 1.2 秒的页面，用户的留存数据可能差异显著。分数线只是及格线，不是满分线。

## 六、优化起点：测量

没有测量数据的优化是盲目的。推荐的定位路径：

1. **Chrome DevTools → Lighthouse**：拿到 LCP/FCP/TBT/CLS 的基线数据和优化建议
2. **Performance 面板录制**：找到阻塞主线程的长任务，分析调用栈
3. **Network 面板**：检查关键资源加载时序，确认是否有被阻塞或被降级的资源
4. **webpack-bundle-analyzer**：确定产物构成，识别体积异常模块
5. **Web Vitals 库（`web-vitals`）**：在 RUM（Real User Monitoring）中采集真实用户的 CWV 数据，不同于 Lab 数据，RUM 反映的是不同设备、不同网络条件下用户的真实体验分布
